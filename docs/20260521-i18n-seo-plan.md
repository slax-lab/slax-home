## Context

Slax 官网（`slax.com`，Astro 6 + MDX，单仓部署 Cloudflare Pages）目前是纯英文站，零 i18n 基础设施：

- `<html lang="en">` 写死，无 hreflang，无 sitemap 多语言节点
- 17 个 `.astro` 页面 + 76 条内容集合条目（blog 8 / reader-changelog 20 / note-changelog 36 / reader-alternatives 7 / note-alternatives 5），全部英文 kebab-case slug
- 三个产品着陆页（index/reader/note）合计 3000+ 行硬编码英文文案
- OG 渲染器只加载 Latin 字形，无 CJK 字体
- 已有 sitemap、JSON-LD（WebSite + BreadcrumbList）、canonical 等基础 SEO 设施

参考 inoreader.com 的 i18n 模式（subdirectory `/{lang}/`、英文留 `/`、完整 hreflang 互链 + `x-default`、无 Accept-Language 自动跳转），改造为支持 **5 种语言**（en/zh-Hans/zh-Hant/ja/ko）的国际化站点，**所有现存英文 URL 保持不变**，新增 4 个语言版本走 `/{lang}/...` 前缀。

目标：在不丢失 SEO 权重、不破坏现有外链的前提下，把全站（含 blog/changelog）多语言化，并补齐多语言相关的 SEO 最佳实践（hreflang、sitemap-i18n、canonical、`inLanguage` 结构化数据、本地化 OG）。

**参考实现**：`/Users/wulujia/Dropbox/Github/sgai/`（也是 Astro 多语言站，已上线 zh/en/ja）。架构方向高度一致，可直接借鉴的代码点：

- `src/i18n/index.ts` 的 `unprefixed(path)` 函数（剥离当前路径首段 lang 前缀）—— 用来给 hreflang 生成 reverse-map
- `src/components/CommonMeta.astro` 的 hreflang 块生成逻辑 —— 完整可抄
- `src/utils/blog.ts` 的 lang-from-path 推断模式 —— 我们的动态路由 `getStaticPaths` 直接对照
- `package.json` 的 `check:i18n-completeness`、`check:i18n`、`i18n-pair` 等 npm scripts —— 验证翻译配对完整、catch 孤立 slug

**有意不学的**：sgai 因历史原因把"路由默认语言（en）"和"数据字段默认 locale（zh）"拆成两个常量，又因路由重复问题关闭了 Astro 原生 `i18n` 配置块自己手写路由。Slax 从零开始，**用 Astro 6 原生 `i18n` 配置 + 单一 defaultLocale=en**，更干净。

## 架构决策

### 1. URL 与路由

- `en` 是默认 locale，**不加前缀**，所有现存 URL 不动
- 其他 4 种语言走 `/{lang}/...`：`/zh-Hans/`、`/zh-Hant/`、`/ja/`、`/ko/`
- 用 BCP-47 标准码（`zh-Hans` 不用 `zh-CN`，`zh-Hant` 不用 `zh-TW`），更符合 Google hreflang 规范且地域中立
- **不做** Accept-Language 自动跳转（对 CDN 缓存和 Google 爬取最友好；用户用语言切换器主动切）
- 直接访问 `/` 永远返回英文（HTTP 200，无 redirect）

**Astro 6 i18n 配置**（`astro.config.mjs`）：

```js
i18n: {
  defaultLocale: 'en',
  locales: ['en', 'zh-Hans', 'zh-Hant', 'ja', 'ko'],
  routing: {
    prefixDefaultLocale: false,
    redirectToDefaultLocale: false,
    fallbackType: 'redirect'
  },
  fallback: { 'zh-Hant': 'zh-Hans' }  // 繁体未翻译时回退简体（路由级）
}
```

### Fallback 策略（两层）

- **路由 fallback**（Astro 原生）：`zh-Hant` 缺该页 → 301 到 `zh-Hans` 对应页；其他语言缺页 → 301 到英文（用 Astro 6 默认行为）
  - 注意：fallback 后**该 URL 不进 hreflang/sitemap**，避免被 Google 当成重复内容
- **字典 fallback**（`t()` 工具函数内）：UI 字符串缺 key 时按链回退：
  - `zh-Hant` → `zh-Hans` → `en`
  - `ja` / `ko` → `en`（不互相回退，避免日韩交叉）
  - 缺到底（连 `en` 都没有）→ 抛错，build 阶段就暴露

### 2. 页面文件组织

保留现有 `src/pages/` 作为英文版（无需迁移），新增 `src/pages/[lang]/` 子树。每个英文页面在 `[lang]` 子树有对应文件，**共享一个 PageShell 组件**避免拷贝：

```
src/pages/
├── index.astro              [EN /]
├── about.astro              [EN /about]
├── reader/index.astro       [EN /reader/]
├── blog/[slug].astro        [EN /blog/:slug]
├── [lang]/
│   ├── index.astro          [/{lang}/]
│   ├── about.astro
│   ├── reader/index.astro
│   ├── blog/[slug].astro
│   └── ...（与英文树对称）
```

**避免重复**：把每个页面的 JSX 抽到 `src/components/pages/*.astro`，`src/pages/*.astro` 和 `src/pages/[lang]/*.astro` 都只调用该组件、传入 `lang` prop。

`getStaticPaths` 模式（动态 `[lang]` 段）：

```astro
export async function getStaticPaths() {
  return ['zh-Hans', 'zh-Hant', 'ja', 'ko'].map(lang => ({ params: { lang } }));
}
```

### 3. 内容集合（Content Collections）

**目录式按语言分桶**（每个 collection 内新建语言子目录）：

```
src/content/blog/
├── en/
│   ├── slax-reader-cli.mdx
│   └── ...
├── zh-Hans/
│   ├── slax-reader-cli.mdx   ← 同名 slug = 翻译对
│   └── ...
└── ja/...
```

`src/content.config.ts` 改造：

- 5 个 collection 全部启用 glob loader（已是）
- 在 schema 加：
  - `lang: z.enum([...]).optional()`（不写则从路径推断）
  - `translationOf: z.string().optional()`（默认用 slug 自动配对，特殊情况手动指定）
- 在每条 entry 解析时从 `id` 路径首段提取 lang
- 现有英文 mdx 文件**全部移到 `en/` 子目录**（一次性 `git mv`，URL 不变因为 slug 计算逻辑去掉首段 lang）

**slug 计算逻辑**（在动态路由 `getStaticPaths` 里）：

```ts
const posts = await getCollection('blog');
return posts.map(post => {
  const segs = post.id.split('/');
  const lang = segs[0];                       // 'en' | 'zh-Hans' | ...
  const slug = segs.slice(1).join('/').replace(/\.mdx$/, '');
  return { params: { lang: lang === 'en' ? undefined : lang, slug }, props: { post } };
});
```

**缺译策略**：某 slug 只在 EN 存在 → 其他语言版本不生成路由也不进 sitemap。**不做内容回退**（避免给用户假装翻译过的体验，也避免 Google 把英文内容当成 zh 内容索引）。

### 5. SEO Meta：Base.astro 改造

`src/layouts/Base.astro` 必须新增 props 和 logic：

- 新增 props：`lang`（必填）、`alternates`（可选，`{ [lang]: path }`，用于内容页指定翻译对的精确路径）
- `<html lang={lang}>` 改为动态
- 生成 hreflang：遍历所有 locales，对每个有翻译的版本输出 `<link rel="alternate" hreflang="{bcp47}" href="{absUrl}">`，最后加 `<link rel="alternate" hreflang="x-default" href="{enUrl}">`
- canonical 指向当前语言版本自身（不指向英文）
- JSON-LD WebSite/Article 加 `inLanguage` 字段
- 字体 preload 按 `lang` 切换：英文 + 拉丁语料用 Latin 子集；CJK 语言加载 Noto Sans SC/TC/JP/KR 对应子集
- Twitter Card 补 `twitter:title` / `twitter:description`（当前缺失）

**辅助函数**（新建 `src/lib/hreflang.ts`）：

```ts
buildAlternates(currentPath: string, currentLang: Locale, available: Locale[]): Alternate[]
```

输入当前页面在英文下的路径 + 该页面有哪些语言版本，输出全套 `<link rel="alternate">` 数据。

**Hreflang 生成算法**（直接借鉴 sgai `src/components/CommonMeta.astro` 和 `src/i18n/index.ts` 的 `unprefixed()`）：

```ts
// src/lib/hreflang.ts
const LOCALES = ['en', 'zh-Hans', 'zh-Hant', 'ja', 'ko'] as const;
const DEFAULT_LOCALE = 'en';

// 剥离路径首段 lang 前缀，得到"locale-agnostic 路径"
export function unprefixed(pathname: string): string {
  const segs = pathname.split('/').filter(Boolean);
  if (segs[0] && LOCALES.includes(segs[0] as any) && segs[0] !== DEFAULT_LOCALE) {
    return '/' + segs.slice(1).join('/');
  }
  return pathname;
}

// 把 locale-agnostic 路径加回某语言前缀
export function withLocale(path: string, locale: Locale): string {
  const clean = path.startsWith('/') ? path : `/${path}`;
  return locale === DEFAULT_LOCALE ? clean : `/${locale}${clean}`;
}

// 生成完整 alternates 列表
export function buildAlternates(
  currentPath: string,
  available: readonly Locale[] = LOCALES
): Array<{ hreflang: string; href: string }> {
  const stem = unprefixed(currentPath);
  const out = available.map(lang => ({
    hreflang: lang,
    href: new URL(withLocale(stem, lang), SITE).toString()
  }));
  out.push({ hreflang: 'x-default', href: new URL(withLocale(stem, DEFAULT_LOCALE), SITE).toString() });
  return out;
}
```

`Base.astro` 调用：

```astro
---
const alternates = buildAlternates(Astro.url.pathname, Astro.props.availableLocales ?? LOCALES);
---
{alternates.map(a => <link rel="alternate" hreflang={a.hreflang} href={a.href} />)}
```

**关键约束**：
- `availableLocales` 由调用方传入，**只列已翻译的语言**（缺译不出现在 hreflang）
- 静态页面（about/privacy/terms 等）默认认为 5 种全有
- 内容页面（blog/changelog/alternatives）在 `getStaticPaths` 里根据"同 slug 在哪些语言目录存在"动态决定
- 所有语言的 hreflang **必须对称回链**（A 列了 B，B 就必须列 A），否则 Google 会忽略整组

### 6. Sitemap

`@astrojs/sitemap` 已支持 `i18n` 选项，开启即可在每条 `<url>` 下生成 `<xhtml:link rel="alternate" hreflang="...">` 子节点。**HTML hreflang + sitemap hreflang 双保险**（inoreader 只做了前者，我们做齐）。

```js
sitemap({
  i18n: {
    defaultLocale: 'en',
    locales: { en: 'en', 'zh-Hans': 'zh-Hans', 'zh-Hant': 'zh-Hant', ja: 'ja', ko: 'ko' }
  },
  filter: (page) => !page.includes('/admin/')
})
```

注意：sitemap 集成对 URL 的语言推断是基于路径前缀的，需要保证 `/{lang}/...` 模式严格一致。

### 9. 翻译工作流 + 校验脚本

## Evals 框架（验证体系）

i18n 改造最容易出的 bug 是"看上去对，机器一查全是问题"——hreflang 不对称、canonical 指错、缺译跳到 404、UI 撑爆容器、翻译质量参差、字体回退掉到系统默认。所以 evals 不是上线之后的事，是**每个 phase 的退出门槛**。

参考 sgai 的 `check:i18n-completeness` / `eval:i18n` / `eval:translation` 三层脚本结构，针对 Slax 设计 4+1 层 evals：

### 10.1 Build-time static checks（必过，CI 强制）

打包成 `pnpm check:i18n`，pre-push hook + Cloudflare Pages 构建前都跑：

1. **翻译配对完整性** (`scripts/i18n/eval-completeness.ts`)
   - 扫每个 content collection 的 (slug, supported_langs) 矩阵
   - 警告：英文版 `pubDate` 超过 30 天但翻译缺失
   - 错误：UI 字典任何非 en locale 缺 en 已有 key

2. **hreflang 对称性 + 合法性** (`scripts/i18n/eval-hreflang.ts`)
   - 解析 `dist/**/*.html` 的 `<link rel="alternate">`
   - 构造有向图 A→B（A 页 hreflang 列了 B），验证**对称闭包**：缺一对就报错
   - 验证每个 hreflang lang code 是 BCP-47 合法（不允许 `zh-CN`、`jp` 等错码）
   - 验证 `x-default` 存在且 href 指向 en 版本
   - 验证目标 URL 在 `dist/` 里**实际存在**（防 hreflang 指向 404）

3. **canonical 正确性**（同脚本）
   - 每页 canonical = 自身 URL（绝对路径、含 https、含 site）
   - 不允许任何页面 canonical 指向其他语言

4. **`<html lang>` 正确性**
   - 与 URL 前缀严格对应（`/zh-Hans/...` → `lang="zh-Hans"`）

5. **JSON-LD 结构化数据**
   - 每页 `inLanguage` 字段 = `<html lang>`
   - 用 `schema-dts` 做类型校验，外加抽样跑 Google Rich Results Test

6. **Sitemap 完整性** (`scripts/i18n/eval-sitemap.ts`)
   - 总条目数 = Σ(content_i × translated_langs_i) + static_pages × supported_langs
   - 每条 `<url>` 含 `<xhtml:link rel="alternate" hreflang="...">` 子节点
   - 无 dup `<loc>`

7. **URL 不变性 regression** (`scripts/seo/eval-url-snapshot.ts`)
   - 维护 `tests/url-snapshot.json`：所有英文 URL 的快照（pre-i18n 状态）
   - 新 build 比对：英文 URL **缺失** = 红色 fail；新增需 PR 描述里手动 ack
   - 这是 SEO 权重保护的最关键 eval

8. **字体加载策略** (`scripts/i18n/eval-fonts.ts`)
   - 解析每页 `<link rel="preload" as="font">`
   - zh-Hans/zh-Hant/ja/ko 页面必须 preload 至少一个对应 CJK 字体子集
   - en 页面不应 preload CJK（浪费带宽）

### 10.2 Translation quality evals（LLM judge + 规则）

`pnpm eval:translation`，发布前手动跑（不进 CI 因为慢 + 耗 API 额度）：

1. **Glossary 遵守**（规则，秒级）
   - 扫每篇翻译 `.mdx`，正则匹配保留词（Slax / Slax Reader / Slax Note / iOS / macOS / GitHub / 域名 / 等）
   - 任何保留词在翻译中被本地化 → 报错带行号
   - glossary 表存 `src/i18n/glossary.md`

2. **术语一致性**（规则）
   - 术语映射表：英文术语 → 各语言译法（如 "read later" → zh-Hans "稍后读" / zh-Hant "稍後讀" / ja "あとで読む" / ko "나중에 읽기"）
   - 扫翻译文件，检查同术语全站一致
   - 不一致 → 报警 + 给出建议

3. **LLM judge**（Claude API，按需）
   - 输入：英文原文 + 翻译版本
   - 评分 1-5：准确性、流畅度、品牌语气、SEO 友好（关键词保留）
   - <3 分标红需人工 review
   - 抽样策略：着陆页 + about 必跑全文；blog/alternatives 抽 20%；changelog 抽 5%

4. **长度合理性**（规则）
   - title 翻译后 ≤ 60 字符（CJK 按 2 字符宽度计） → 防 SERP 截断
   - meta description ≤ 155 字符（同上）
   - 超限列表 + 精简建议

### 10.3 SEO evals（外部工具，build 后）

`pnpm eval:seo`，对 dev preview 站跑：

1. **Lighthouse**（@lhci/cli）
   - 6 个代表 URL：`/`、`/zh-Hans/`、`/zh-Hant/`、`/ja/`、`/ko/`、`/blog/<random>`
   - SEO ≥ 95、Accessibility ≥ 90，否则 fail
   - 报告写 `evals/lighthouse/<timestamp>/`

2. **hreflang validator**（https://technicalseo.com/tools/hreflang/ 或本地实现）
   - 抽样 10 个 URL，全部 PASS
   - 已在 10.1 #2 做本地校验，这里是双保险用外部工具复核

3. **Rich Results Test**（Google API）
   - 抽样 5 个 URL，验证 WebSite/Article 结构化数据被识别

4. **OG 卡片抽查**
   - 部分自动化：fetch HTML、解析 `og:image`、HEAD 验证返回 200 + Content-Type
   - 部分人工：每语言抽 1-2 个新链接到 X / LinkedIn / Slack 私信粘贴肉眼看（已在原 CLAUDE.md OG 章节列出，沿用）

### 10.4 Visual regression（Phase 2 起）

`scripts/seo/eval-visual.ts` 用 Playwright：

1. 截图每个 (page × lang) 的 hero + 关键 section
2. 与 baseline 对比，diff > 5% 报警
3. 重点抓：UI 撑爆（zh-Hant 用词可能比 zh-Hans 长 10-15%）、字体回退（缺 CJK preload 掉系统字体）、布局错位
4. Phase 1 不阻塞（壳子页面简单），Phase 2 完成时建立 baseline

### 10.5 Live monitoring（post-deploy，持续）

不进 build，但要建立监控：

1. **Cloudflare Analytics**：监控 404 / 5xx，按语言分类
2. **GSC**：每周脚本拉 Coverage 报告，发现 hreflang error / soft-404 报警
3. **Indexing rate 监控**：通过 `site:slax.com/zh-Hans/` 跑定时查询，估算各语言 indexed 比例
4. **季度 review**：流量 / 排名 / CTR 按语言对比；判断是否值得继续维护某低活语言

### Evals 的"假阳性 + 假阴性"对策

- **假阳性多**（误报）→ 团队会无视 → 必须保证报错率 < 5%。对策：
  - 静态检查严格 0 容忍（必过）
  - LLM judge 阈值留出余地（<3 分才红，3-4 分黄色仅提示）
  - URL 不变性允许"经手动 ack 的新增 URL"白名单
- **假阴性多**（漏报）→ 上线才发现 → 加 live monitoring 兜底
  - 部署后 24h 内自动跑一遍 GSC URL Inspection（抽样）
  - 用 Slack webhook 给 Luca 推送异常

## 风险与备注

- **繁简关系**：zh-Hant 第一版用 zh-Hans 的转换（OpenCC）+ 术语词替换打底，再补香港/台湾本地化用词（"網誌"、"通知"、"設定"等）。Google 把 zh-Hans 和 zh-Hant 当作不同语言。
- **日文/韩文质量**：如果无母语校对资源，宁可少翻几页质量稳一些，也不要全 LLM 直翻——劣质翻译会让 Google 给该语言整体降权。
- **OG 图体积**：5 lang × 全站 OG 可能数百张 PNG。Cloudflare Pages 静态资源费用低，但 build 时间可能从分钟级涨到 10+ 分钟，需关注。
- **URL 不变性是底线**：所有现存英文 URL 必须 100% 保持，不能有任何 redirect 链路。新增 `[lang]` 段绝不影响英文路由。
- **Search Console 老数据**：英文版数据不会重置；新增语言版本的数据需要 2-4 周开始有趋势。
