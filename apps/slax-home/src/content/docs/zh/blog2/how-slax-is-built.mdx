感觉简单的产品，构建起来很少是简单的。你希望用户体验到的摩擦越少，工程层面需要承担的复杂度就越多。

Slax 是一款稍后读与笔记工具。听起来直截了当——抓取 URL、干净地渲染、让用户标注。但细节会迅速叠加：

- 文章来自成千上万个 HTML 结构各异的网站。
- 即使源文章被编辑后，高亮也需要锚定在正确的文字上。
- 在 Web、iOS 和桌面端之间，同步需要既快速又无冲突。
- 离线阅读必须可用，包括刚保存的新文章。

以下是我们处理每个问题的方式。

## 内容提取：解析混乱的网络

第一个挑战是将任意 HTML 转化为干净可读的内容。我们使用了以下组合：

1. **基于 Readability 的提取** — 一种受 Firefox 阅读模式启发的技术，通过内容密度对 DOM 节点打分，并提取正文主体。
2. **高价值域名的自定义解析器** — 像 Twitter、Substack 和主流报刊这类网站的 HTML 足够稳定，特定域名规则能带来更好的效果。
3. **降级渲染** — 当提取失败或置信度较低时，我们渲染原始页面的沙箱 iframe，而不是展示乱码文本。

最难处理的是付费墙内容。我们从预览内容中尽可能提取，并清晰标明付费墙的起始位置，而不是假装文章是完整的。

## 高亮锚定

高亮使用**文本位置模型**而非原始 DOM 节点路径来锚定。这意味着我们记录：

- 选中内容前的 32 个字符
- 选中的文字本身
- 选中内容后的 32 个字符

当你重新打开一篇文章时，高亮引擎扫描文本，匹配这个模式，并重新附着高亮——即使文章 HTML 有轻微变化。

```typescript
interface HighlightAnchor {
  prefix: string;    // 选中内容前 32 个字符
  exact: string;     // 高亮的文字
  suffix: string;    // 选中内容后 32 个字符
  textOffset: number; // 回退用的字符位置
}
```

这比基于 DOM 的锚定更健壮（DOM 锚定在页面重新渲染时会断开），也比精确字符偏移量对小幅编辑更有弹性。

## 同步架构

Slax 为笔记使用受 **CRDT 启发的同步模型**。与最后写入获胜不同，每个笔记操作都被存储为只追加的事件日志：

```
[创建笔记]
[添加段落: "初始文字"]
[编辑段落 1: "修订文字"]
[添加标签: "效率"]
```

同步时，我们确定性地合并这些事件日志。两台设备在离线状态下做出不同修改，同步后总会产生相同的最终状态，没有冲突，没有数据丢失。

对于高亮和阅读位置，我们使用更简单的 LWW（最后写入获胜）语义，因为这些操作的重要性相对较低。

## 离线优先

每个 Slax 客户端都在本地缓存最近 200 篇文章的全文。当你保存新文章时，它会立即加入后台下载队列。

结果：在飞机上打开 Slax 和在 WiFi 下打开的体验完全相同。你可以在没有网络连接的情况下阅读、高亮和标注，一切在重新联网后同步。

我们在 Web 端使用 Service Worker，在移动端使用原生 SQLite 进行本地存储。同步队列具有持久性——在应用重启后仍然存在，并会持续重试直到操作完成。

## 开放的部分

我们相信数据处理方式的透明度。Slax 的核心解析库——内容提取引擎——在 [github.com/slax-lab](https://github.com/slax-lab) 开源。你可以检查我们从任何文章中精确提取了什么以及如何提取的。

高亮锚定算法和 CRDT 同步协议也在我们的工程 Wiki 中有详细文档，供对实现细节感兴趣的人参考。

---

构建思维工具意味着对可靠性的极度关注。一条消失的高亮、一篇未能同步的笔记、一篇渲染成乱码的文章——每一个问题都会打破那份信任，而这份信任正是第二大脑工具值得使用的根基。

这是我们每天都在解决的工程问题。
