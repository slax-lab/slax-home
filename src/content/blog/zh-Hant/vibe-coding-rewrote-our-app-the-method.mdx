這份總結基於多個群聊、方案預研文件、3 月 18 日的會議紀錄，以及 3 月中旬到 4 月中旬多篇日報整理而成。

目標不是做技術流水帳，而是回答兩個問題：

1. 從 2026-03-10 之後，我用 AI coding 重寫 [Slax Note](https://slax.com/note/) 的心路歷程是什麼？
2. 這次實驗裡，哪些經驗值得以後複用，哪些坑必須提前規避？

## 一、先說結論

這次 AI coding 重寫 Slax Note，不是一次「AI 自動寫完 App」的輕鬆勝利，而是一段從興奮、懷疑、受挫、調整方法，到重新建立信心的過程。

用一句話概括：

> AI 確實把開發門檻打下來了，但真正決定效率的，不是「會不會用 AI」，而是「會不會定義問題、約束上下文、管理品質、建立協作機制」。

這次重寫最大的價值，不只是做出了一個 iOS 新版本，而是驗證了幾件事：

- 產品/設計師確實可以借助 AI 深度參與甚至主導實作，不再只是提需求和驗收
- AI 很適合高速搭框架、補介面、做中小型修復和批次試錯
- 一旦進入真實業務鏈路、真機環境、複雜同步和多端協作，AI 就不會自動幫你收口，反而會暴露出對規格、上下文、驗證、程式碼 review 的更高要求
- 後期真正拉開差距的不是「寫程式碼」，而是「發現根因、控制回歸、沉澱方法」

這次實驗的本質，是把原本隱性的研發能力，拆成了可以由人 + AI 協作完成的一套流程。

## 二、為什麼會開始這次重寫

如果把時間線拉早一點，這件事的起點不是 3 月中下旬，而是 2026-03-10 左右就已經明確浮出水面。

當時團隊討論得已經很直接：想拿 Slax Note 來做一次更激進的 AI coding 試驗，甚至有「把 Slax Note 的程式碼全部刪掉，由更小團隊借助 AI 重做和維護」的設想。這個起點本身就說明，當時討論的已經不是「用 AI 提高一點效率」，而是在試探一種新的產品開發方式。

從 3 月中旬開始，團隊逐漸不再把 AI 看成單點提效工具，而是認真討論一種更激進的開發方式：

- 產品確定需求
- 不再層層中轉給研發實作
- 而是由產品/設計師直接借助 AI 生成程式碼、除錯、驗收
- 甚至後續 review、修 bug 也盡量借助 AI 完成

但這裡有一個非常關鍵的前提：

> 這次重寫不是「直接把需求丟給 AI 讓它從零亂寫」，而是一次 spec 驅動的 AI coding。

在正式重寫前，研發工程師玉潔先基於已有程式碼倉庫，把原有產品裡的關鍵結構、能力邊界、頁面邏輯、介面契約、模組關係等內容，提煉成了一組可供 AI 消費的 spec 檔案。後續的開發，不是讓 AI 自由發揮，而是讓它圍繞這些 spec 分步執行。

這一點非常關鍵，它解釋了這次實驗為什麼不是碰運氣：

- AI 不是憑空生成，而是在消費工程化過的上下文
- 產品/設計師不是直接面對一整個複雜倉庫，而是在面對一組結構化任務說明
- 後續遇到問題時，也不是回到「純口頭描述」，而是可以回到 spec 和文件繼續修正

換句話說，這次實驗真正驗證的，不是「有了 AI 就不需要研發」，而是：研發可以先把自己的經驗、架構理解和系統知識沉澱成 spec，再讓更多人借助 AI 進入實作層。

## 三、這次重寫為什麼能推進：關鍵協作支撐

如果只看表面，很容易把這次過程理解成「產品 + 設計師 + AI 把 App 重寫出來了」。

但更準確的說法是：這次重寫的推進，是建立在研發同事、後端同事和 AI 協作的多方支撐之上的。

### 1）研發工程師先把複雜系統抽成 spec

這是最核心的一步。

玉潔先根據既有程式碼倉庫，把原本散落在程式碼裡的產品結構、模組關係、頁面邏輯、介面資訊和實作約束，提煉成了十幾個 spec 檔案。這樣後續的 AI coding 才不是「盲寫」，而是變成一種按 spec 分步執行的過程。

這一步的價值在於：

- 降低 AI 幻覺和亂改的機率
- 降低非研發角色直接面對複雜工程的門檻
- 讓開發過程從「憑 prompt 試錯」變成「圍繞規格執行」

### 2）研發同事把隱性的工程經驗，轉成文件化規則

除了 spec，本次推進裡還有很多關鍵經驗並不是寫在需求文件裡的，而是同事們在實踐中不斷補上的，例如：

- 某些 UI/互動在當前技術棧下的實作邊界
- 哪些能力適合先簡化實作，哪些地方不能硬剛
- 某些介面欄位、命名方式、狀態處理的真實約束
- 哪類問題應該優先查根因，哪類問題可以先跳過

這些內容一旦被寫進文件，AI 的穩定性和團隊協作效率就會明顯提高。

### 3）後端同事的配合貫穿全程

除了前端/客戶端側的 spec 提煉、實作推進和問題排查，這次重寫能持續推進，還有一個非常關鍵的支撐力量：後端同事姚坤的密集配合。

**早期：參與 prompt、熱詞提取和多語言支援。** 在 2026-03-10 當天，姚坤就已經在調整提取專用詞的 Prompt、優化對葡萄牙語和英文的支援、和我們討論熱詞提取、處理速度、可接受性等問題。這說明他一開始參與的，不只是後端介面，而是產品能力本身的 AI 表達和效果優化。

**中期：持續補介面、遷移介面、補充 API 文件。** 隨著重寫進入真實鏈路，他的支援逐漸擴展到遷移和補充介面、持續補齊 API 文件、幫助大家理解真實的服務端約束。AI coding 要想進入真實業務，不可能只靠前端自己猜介面。

**後期：把 API 文件 / Apifox / MCP 一起接進 AI 工作流。** 到後面，他的支援已經不只是「答疑」了，還進一步演化成：讓大家對照 Apifox 文件讓 AI 分析問題、提醒先刷新文件快取再按介面規約排查、提供 Apifox MCP 的設定方式、直接指出參數問題、重複請求、請求體異常等具體問題。

> 這次重寫的順利推進，不只是「AI 在寫程式碼」，而是人類同事在不斷給 AI 提供正確的邊界、上下文、規約和回饋。

### 4）這次實驗的成功，本質上是一次跨角色協作成功

回頭看，這次重寫之所以能真正跑起來，不是因為某一個角色單獨變強了，而是因為：

- 研發工程師把已有倉庫經驗提煉成 spec
- 產品 / 設計師借助 AI 推動實作和驗證
- 後端同事持續補介面、補文件、補互動細節、補定位能力
- 大家一起把問題從「AI 會不會寫」推進到「產品能不能真的跑通」

更準確的說法是：這次重寫是一次「spec 驅動 + 文件驅動 + 前後端密切配合 + 人機協作」的綜合實踐。

## 四、心路歷程：這一個多月裡，我怎麼一步步走過來的

### 1）啟動期：新鮮、興奮，但也帶著明顯的不確定感

2026-03-10 到 2026-03-18 這段時間，整體情緒是興奮的，但不是篤定的。

一方面，團隊一邊做準備工作，一邊統一認知：設計師開始安裝和學習 Claude Code；大家討論 AI 重寫到底要做到什麼程度；也在討論原生、跨平台、雲端環境、本機環境這些路徑選擇。

另一方面，我自己對這件事的第一反應其實也很真實：

- 「變天了家人們」
- 「我連版本控制都不會，你讓我來維護一個產品」
- 「就離譜」

這些話很能說明當時的狀態：不是一開始就確信這條路一定能成，而是既興奮又震驚，既想試又覺得這事太激進。

這個階段最重要的變化，不是程式碼本身，而是心態上的切換：原本「實作」是研發的地盤，但 AI 讓「實作」第一次對產品和設計師開放了。

2026-03-18，第一版基於文件跑出來的 demo 已經有了輪廓。那時的感覺更接近：

- 這條路不是幻想
- AI 不是只能寫玩具 demo
- 只要文件、上下文、目標說清楚，它確實能把一個產品介面和主流程快速搭出來

但也正是在這裡，後面的難點其實已經埋下了：AI 最擅長的是「把東西做出來」，但不等於「把東西做對、做穩、做成產品」。

### 2）推進期：從「圍觀 AI 寫程式碼」進入「真實開發現場」

到 2026-03-20 到 2026-03-23，狀態明顯變了。

前幾天更多是「它能不能做」，這幾天開始變成「它做出來的東西到底能不能用」。這時候我和團隊碰到的，不再是抽象問題，而是很具體、很磨人的現實：

- UI 細節改不準
- 模擬器看起來沒問題，真機就出問題
- 接後端介面後，效能、互動、狀態管理問題一口氣冒出來
- 一個地方修好了，另一個地方又壞掉
- AI 明明說已經改好了，但效果和預期還是差很遠

我在 2026-03-23 那篇反思裡其實已經把核心痛點說得很透了。

**第一層受挫：問題開始連鎖反應。** 表面看是一個互動細節，背後卻可能牽連手勢衝突、頁面效能、導覽方式、ScrollView / List 取捨。AI 不是不會改，而是它經常會給出一個「能交差但不夠對」的局部解。

**第二層受挫：自己不懂底層，就無法點撥 AI。** 我開始意識到，最大的問題不是 AI 不努力，而是當自己不理解 iOS 機制時，AI 給出的方案到底是最優解、次優解，還是臨時糊上去的解，自己沒有判斷力。

**第三層認知升級：有些需求不能硬剛。** 進入 AI coding 現場後，我開始真正理解研發常說的「要調整實作方式」是什麼意思：有些需求在產品上看合理，在當前技術路徑上卻極不經濟；有些細節繼續死磕，只會陷入低效循環。

### 3）調整期：開始形成自己的方法，不再盲目硬扛

到了 3 月下旬，狀態開始從「被問題推著走」，慢慢變成「我知道該怎麼和 AI 協作」。幾個關鍵轉折非常明顯：

**轉折一：開始接受「不是所有問題都該當場解決」。** 改了很多次還不對的，先跳過；先保主流程，先保可展示，先保真機可跑。AI coding 最大的風險之一就是：你會被「它好像馬上就能改好」的錯覺拖進無窮微調。

**轉折二：開始把經驗寫進文件，而不是只留在對話裡。** 光靠聊天上下文不夠，光靠臨場描述也不夠，要把約束寫成文件，AI 才能越來越穩定。不是每次重新教 AI，而是把規則沉澱成資產。

**轉折三：開始理解「工具熟練度」也是生產力。** 很多問題其實不是程式碼本身，而是切錯資料夾、Xcode 編譯舊檔案、英文報錯看不懂、上下文過長不敢切視窗。這類問題說明：AI coding 的門檻，不只是「會不會提需求」，還包括「會不會使用整套開發環境」。

### 4）收斂期：從「能跑」走向「能用」，從個人試驗走向團隊協作

4 月初到 2026-04-10，是另一個明顯轉折。這時專案已經不再是 demo 階段，而是在往真實交付推進：現場展示、真機測試、TestFlight、團隊試用、多人回饋。

這個階段最大的變化是：問題的重心從「頁面沒做完」，切到了「真實使用下的問題暴露」。

**Bug 數量突然激增。** 一旦推到更多同事手裡，問題就會集中爆發。我在 2026-04-10 意識到：必須把 Bug 管理從「文件紀錄」升級為「結構化台帳」。於是 Bug Tracker 從長文件遷移到多維表格，專案管理方式開始真正升級。

**開始區分「可展示」和「可發布」。** 前期 demo 看起來已經有 80% 功能，甚至可以現場展示；但到真正要發 TF、上架、給更多人使用時，暴露的問題完全是另一個量級。AI 很容易幫你衝到「看起來能用」的階段，但距離「穩定上線」還隔著完整的工程品質關。

**團隊協作方式也在升級。** 不再只是「讓 AI 改一下」，而是開始進入 PR、Review、Code Review、自動部署、修復閉環。我也明確提出：使用 CC 編碼、用 Codex review 程式碼。真正可擴展的 AI coding，不是「靠一個模型從頭寫到尾」，而是讓不同 AI 在不同環節分工協作。

### 5）衝刺期：最難的不是開發，而是系統性問題的根因定位

2026-04-13 到 2026-04-17，是一次很典型的「工程深水區」。我們解決的問題，已經不再是淺層 UI 或單頁面邏輯，而是系統性問題：

- Token 401 級聯清除導致所有同步靜默失敗
- 會話同步欄位類型錯誤、命名方式錯誤、重複同步路徑
- audioFileId 類型不匹配導致 PUT note 一直失敗
- 上傳競態把成功狀態覆蓋回 recognizing
- SwiftUI `.refreshable` 取消任務導致連續失敗
- 背景任務時長不足導致 ASR 卡死
- Android 封存狀態在 iOS 端無法同步

這一階段很容易讓人沮喪，因為它會讓你感覺：每修一個問題，背後都牽出一串機制問題；表面是一個 bug，底層卻是整個架構假設不成立；AI 雖然能幫忙定位，但要想真正修透，仍然需要極強的問題拆解和驗證能力。

走到這裡時，我們已經不只是「會讓 AI 寫程式碼」，而是逐漸具備了：看日誌、找根因、做最小修復、控制回歸、做 PR 說明、建立 fix 閉環。

前期證明了 AI 能幫你「寫出東西」，後期證明了我們正在學會「把東西修成產品」。

## 五、這次過程中最真實的心理變化

如果不寫技術細節，只寫心路歷程，可以概括成下面幾個階段：

1. **最開始：** 很興奮，但帶著不確定感。過去必須依賴研發才能推進的事情，現在自己可以直接下手。AI 不是玩具，而是真的能產出介面和功能。但與此同時，這件事又顯得過於激進，甚至帶著一點「這也行？」的震驚。

2. **很快：** 開始懷疑。一旦進入介面、真機、複雜狀態、細節還原，問題就遠超預期。AI 說得很自信，但結果常常不夠好。開始懷疑：這條路到底是不是高估了？

3. **中段：** 最挫敗。真正挫敗的不是 bug 多，而是自己不懂底層，所以沒法像工程師那樣點撥 AI。只能不斷測試、回饋、再測試。這是這次實驗心理成本最高的一段。

4. **後來：** 開始調整方法。不再要求 AI 一步到位，而是開始做：分階段目標、先主流程後細節、先可展示後可發布、把規則寫進文件、把 bug 收口到台帳、把 review 也交給 AI 參與。

5. **到現在：** 重新建立信心，但不再天真。只要流程設計對、規格寫得清、驗證機制夠嚴，AI 確實可以把很多原來做不到的事情做成。

## 六、這次最值得覆盤的經驗教訓

### 1. 不要把 AI 當「自動程式設計師」，要把它當「高執行力但需要約束的協作者」

AI 的特點很鮮明：執行快、產出多、願意不斷重試、會給出看似完整的解釋。但問題也很明顯：容易局部最優、容易在錯誤路徑上越走越遠、容易過度自信、容易「修了這個壞那個」。

更好的用法不是把任務一扔等它交結果，而是：明確目標、收緊上下文、控制變更範圍、每一輪都驗證結果。

### 2. 文件不是輔助物，而是 AI coding 的核心生產資料

這次很明顯驗證了一件事：**AI coding 的上限，很大程度取決於文件品質。**

好的文件會直接提升需求理解準確率、UI 還原穩定性、介面對齊效率、多人協作一致性、新問題的處理速度。反過來，如果文件不清晰，AI 就只能靠猜；而一旦靠猜，返工幾乎不可避免。

文件至少要分三層：

- **產品規格：** 目標、邊界、優先級
- **實作約束：** 命名、介面欄位、狀態機、元件規範
- **經驗文件：** 已經踩過的坑、禁止再試的低效路徑

### 3. 真正耗時的往往不是「寫」，而是「驗」

前期會誤以為 AI coding 的核心挑戰是生成；但做下去就會發現，真正吃時間的是驗證：

- 驗證是否符合預期
- 驗證是否影響別的功能
- 驗證模擬器和真機是否一致
- 驗證多端狀態是否一致
- 驗證是不是只修了表面

AI 把「編碼成本」打下來了，但「驗證成本」反而變得更重要了。

### 4. Bug 一多，必須立刻結構化管理，不能靠聊天和長文件扛

2026-04-10 建立 Bug Tracker 是一個非常正確的動作。當問題規模上來以後，如果沒有結構化管理，就會出現同一個問題反覆討論、優先級混亂、負責人不清、修復狀態不可追蹤、版本對應關係丟失。

進入多人測試階段，就應該立刻有編號、優先級、負責人、目前狀態、重現路徑、所屬版本。

### 5. 「會不會點撥 AI」會決定效率上限

不會寫程式碼，並不代表不能做 AI coding；但如果完全不理解底層機制，就會有一個明顯上限：你只能描述現象，很難判斷方案優劣，很難識別假修復，很難主動調整實作路徑。

兩條路都值得做：

**路線 A：提高非研發角色的工程素養。** 至少要懂日誌怎麼看、檔案關係怎麼看、編譯鏈路怎麼看、常見錯誤怎麼判斷。

**路線 B：讓 AI 補上「解釋層」和「審查層」。** 例如強制要求 AI 輸出根因假設、改動範圍、風險點、驗證步驟、回滾方式。

### 6. 真正高效的 AI coding，不只是 prompt 寫得好

這次後面越來越清楚的一件事是：

> AI coding 真正高效，不是因為 prompt 神奇，而是因為 AI 被放進了一個有規約、有文件、有工具、有人工回饋的系統裡。

我們實際跑通的鏈路，已經不只是「對 AI 說一句需求」，而是：用 spec 定邊界、用文件定約束、用 Apifox / MCP 接入介面規約、讓人工同事持續糾偏和補資訊，再讓 AI 圍繞這些材料執行。

### 7. 先做可用閉環，再追求完美還原

很多痛苦都來自一個很真實的問題：當你還沒有穩定閉環時，過早追求細節，會把專案拖進低效微調。

正確順序應該是：

1. 主流程跑通
2. 真機可用
3. 核心狀態穩定
4. 多人可測
5. 再逐步追 UI / 動效 / 互動精細度

### 8. Review 程式碼不能省，而且越到後期越重要

這次後期很明顯已經進入一個新階段：不只是讓 AI 寫，而是用不同 AI 做 code review、找風險、補測試、做 PR 說明。

這說明一個關鍵事實:AI coding 專案越往後，review 的重要性越高，而不是越低。

如果要規模化，建議把 review 機制前置成預設動作：寫完必 review、關鍵鏈路必做根因說明、關鍵改動必帶驗證方案、高風險修復必控制變更範圍。

## 七、最寶貴的收穫是什麼

最寶貴的收穫，不是「做出了一個版本」，而是我們已經摸到了一套更真實的認知：

1. **AI coding 不是神話，但也不是噱頭。** 它既沒有想像中那麼自動，也絕對不像外界懷疑的那樣只是 demo 工具。

2. **產品、設計、研發的邊界會被重寫。** 不是誰替代誰，而是誰能更快進入問題現場，誰就更有推動力。

3. **未來比拼的不是誰會用 AI，而是誰能建立更好的 AI 協作系統。** 包括文件系統、約束系統、review 系統、bug 管理系統、測試與發布系統。

4. **非研發人員可以深入到實作層。** 以前產品進入實作層，往往會被工具鏈和技術細節擋住；這次雖然很辛苦，但也證明了一件事：你已經不是只能「提需求」的角色，而是可以真正推動實作落地的人。

## 八、給下一次 AI coding 實驗的建議

**啟動前：**

- 先寫清楚目標：是 demo、可展示、TF、還是正式上線
- 先定優先級：主流程 > 穩定性 > UI 細節
- 先準備規範文件、介面文件、設計約束

**開發中：**

- 每天沉澱經驗文件，不要只留在聊天紀錄裡
- 改 2~3 次還不對的問題，及時跳過或換路徑
- AI 每次提交改動時，要求附帶風險點和驗證步驟
- 重要問題必須做根因定位，不能只修表面現象

**進入測試後：**

- 立刻建立 Bug Tracker
- 把問題按 P0/P1/P2 分層
- 一輪只收口最關鍵問題，不貪多
- 開始引入 AI review、PR 規範和回歸驗證

**準備發布時：**

- 把「能展示」和「能發布」明確區分
- 同步、登入、付款、音訊、狀態管理這些鏈路優先級最高
- 一切高頻核心路徑都要真機驗證，不迷信模擬器

如果只看過程，這次重寫其實並不輕鬆，甚至可以說很折騰；但如果看結果，它很值。
