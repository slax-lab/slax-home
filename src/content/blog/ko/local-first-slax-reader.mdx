## 간단한 테스트를 해 보세요

2019년, Ink & Switch는 [*Local-first software: You own your data, in spite of the cloud*](https://www.inkandswitch.com/essay/local-first/)을 발표했습니다. 글 말미에서 엔지니어에게 한 줄 팁을 남겼습니다.

> "Test your app by turning off your WiFi, or using traffic shapers such as Chrome Dev Tools network condition simulator"

Wi-Fi를 끄는 것이 어떤 앱이 진짜로 local-first인지 확인하는 가장 간단한 방법입니다.

저희는 이 방법으로 Read-Later 앱 세 개를 테스트했습니다. 앱을 열고, Wi-Fi를 끄고, 텍스트를 선택해 하이라이트를 눌러 봤습니다.

- **Raindrop**: 하이라이트 메뉴가 나타나지 않습니다.
- **Cubox**: 오류 팝업이 뜹니다.
- **Slax Reader**: 색상이 그대로 나타나고, 상태가 저장되며, 온라인일 때와 차이가 없습니다.

아키텍처가 다르면 결과도 다릅니다.

## Local-first란 무엇인가

Ink & Switch의 논문은 일곱 가지 이상(ideal)을 제시합니다.

| # | Original wording | 한 줄 의미 |
|---|---|---|
| 1 | **No spinners: your work at your fingertips** | 즉각적인 반응, 로딩 표시 없음 |
| 2 | Your work is not trapped on one device | 기기 간 동기화 |
| 3 | The network is optional | 네트워크는 선택사항, 전제조건이 아님 |
| 4 | Seamless collaboration with your colleagues | 실시간 협업 |
| 5 | The Long Now | 회사가 사라져도 계속 작동 |
| 6 | Security and privacy by default | 기본으로 종단 간 암호화 |
| 7 | You retain ultimate ownership and control | 데이터는 사용자의 기기에 존재 |

첫 번째 이상은 이렇게 쓰여 있습니다. "local-first software has the potential to respond near-instantaneously to user input, **never needing to show you a spinner**." 사용자에게 스피너를 보여 주지 않는 것이 말 그대로 1번 이상입니다.

이후 [RxDB 엔지니어들](https://rxdb.info/articles/local-first-future.html)은 *The Local-First Future*를 썼고, 대표 이미지 캡션은 "loading spinner not needed"였습니다. [Spritle](https://www.spritle.com/blog/why-modern-apps-feel-instant-a-simple-guide-to-local-first-development/)의 소개 글은 인상적인 장면으로 시작합니다. "Imagine using your favorite productivity app at 30,000 feet — no internet, no loading spinners, no freezes."

언급되는 제품도 같은 무리입니다. Figma, Linear, Notion, Obsidian, WhatsApp.

## 하이라이트 테스트: Offline-mode는 Local-first가 아닙니다

많은 앱들이 홈페이지에 "even offline"이라고 적습니다. 실제 의미는 한 가지 동작에서 갈립니다.

> **네트워크가 없을 때 여전히 하이라이트하고, 댓글을 달고, 보관할 수 있는가?**

이미 다운로드된 내용을 **읽을** 수만 있고 새로운 무언가를 **쓸** 수 없다면, 그것은 **Offline-mode**입니다.

모든 동작이 온라인일 때와 똑같이 일어나고, 저장되고, 화면에 반영된다면, 그것이 **Local-first**입니다.

차이는 이렇습니다.

| 항목 | Offline-mode | Local-first |
|---|---|---|
| 아키텍처 | Client-Server에 "스냅샷 다운로드" 패치를 추가 | 로컬 DB가 주 저장소, 서버는 동기화 계층 |
| 쓰기 | 네트워크 필요 | 로컬에 즉시 저장 |
| 사용자 체감 | 스피너 / 오류 / 비활성 UI | 대기 없음 |
| 구현 비용 | 낮음 | 높음 |

Offline-mode는 패치이고, Local-first는 아키텍처입니다.

## Read-Later 앱 14종의 실태

저희는 중국어와 영어권 사용자를 대상으로 하는 14개의 Read-Later / 리더 제품을 살펴봤습니다. 홈페이지, 문서, FAQ, 변경 로그를 모두 읽고 실제 Local-first와 Offline-mode 지원 여부를 검증했습니다.

14개 중 자사를 공개적으로 명확하게 local-first라고 칭하는 곳은 Readwise Reader 하나뿐이었습니다.

| 제품 | Local-first | Offline-mode | 핵심 근거 |
|---|---|---|---|
| Readwise Reader | ✅ 지원 | ✅ 지원 | 공식 사이트 원문 "**local-first web app**", 기기 간 동기화 "even offline", 오프라인 전문 검색 가능 |
| Slax Reader(본문) | 부분 ✅ | ✅ 지원 | 하이라이트 / 댓글 / 좋아요 / 보관 모두 로컬에서 완료, 오프라인 경험이 온라인과 동일 |
| GoodLinks | ⚠️ 근접하지만 미주장 | ✅ 지원 | iCloud Sync + 로컬 리치 텍스트 아카이브, 오프라인 읽기 가능; 공식적으로 local-first 표방하지 않음 |
| Instapaper | ❌ | ✅ 지원 | 공식 사이트: "You can read anything you save... **even offline**" |
| Matter | ❌ | ✅ 지원 | App Store: "**Available offline by default**" |
| Wallabag | ❌ | ✅ 지원(Android) | Self-hosted; Android 앱이 오프라인 읽기용으로 이미지 캐싱 지원 |
| Cubox | ❌ | ⚠️ 부분 지원 | 최근 저장한 텍스트는 오프라인 읽기를 위해 자동 캐싱; 이미지 캐시와 오프라인 관리는 "still being improved" |
| Screvi | ❌ | ✅ 지원 | 공식 사이트: "Read anywhere, **even offline**", 모바일에서 오프라인 다운로드 가능 |
| BeeMind | ⚠️ 범위 불명확 | ⚠️ 범위 불명확 | 공식 사이트에 "local-first privacy"와 "local model (for full offline privacy)" 명시, 완전한 정의는 없음 |
| Recall | ⚠️ 브라우징 한정 | ❌ | 공식 사이트: "**Augmented browsing is local first**, your knowledge base is stored safely in the cloud"; offline mode는 사용자 feature request 상태 |
| Raindrop.io | ❌ | ❌ | 공식 FAQ: "**An internet connection is required to sync changes and save new bookmarks**" |
| Karakeep | ❌ | ❌ | 문서에서 자칭 "self-hosting first"; "Offline reading on mobile"은 **Planned** 태그 |
| Noverload | ❌ | ❌ | 자체 비교 페이지가 Offline reading을 자사가 아닌 경쟁사에 부여 |
| Markwise | ❌ | ❌ | 공식 블로그: "**No offline reading mode (yet)**" |
| Readless | ❌ | ❌ | 클라우드 기반 AI 요약 서비스, 오프라인 / local-first 관련 주장 없음 |

표에서 확인된 몇 가지 사실:

- 14개 제품 중 1개가 명시적으로 local-first를 표방하고(Readwise), 1개는 사실상 그렇지만 표방하지 않습니다(GoodLinks).
- 7개는 스냅샷 방식의 오프라인 읽기를 지원합니다. 읽기는 되지만 쓰기는 안 됩니다.
- 4개는 둘 다 지원하지 않으며, 사용량이 가장 많은 Raindrop도 여기에 포함됩니다.
- Wallabag과 Karakeep은 self-hosted입니다. 서버를 자기 집에 둘 수는 있지만, 서버는 여전히 필수 중간자입니다. Self-hosted는 Local-first가 아닙니다.

## 두 가지 배경

왜 Read-Later 카테고리에서 하이라이트 테스트를 통과하는 제품이 이렇게 적을까요? 두 가지 배경이 있습니다.

### 속도의 물리적 한계

Ink & Switch 논문에는 물리에 관한 한 줄짜리 논거가 있습니다.

> "Depending on where you live, the server may well be located on another continent, so **the speed of light places a limit on how fast the software can be**."

5G든 광섬유든, 모든 요청은 서버까지 왕복합니다. 서버가 다른 대륙에 있으면 지연 시간에는 물리적 하한이 존재합니다. 빛이 그 거리를 가로지르는 데 걸리는 시간만큼은 반드시 걸립니다. 대역폭은 늘릴 수 있지만, 지연 시간은 빛의 속도를 넘을 수 없습니다.

RxDB 글은 이 점을 더 구체적으로 풀어 둡니다.

> "Bandwidth Has Grown, But Latency Is Capped... we can't significantly reduce the base round-trip time for each request."

다시 말해, 네트워크 하드웨어가 아무리 발전해도 **서버를 주 저장소로 쓰는 모든 앱은 응답 속도에 넘을 수 없는 천장이 있습니다**.

### 사용자 기대치는 변하고 있습니다

RxDB 글에는 사용자 기대치 변화에 대한 날카로운 논거도 있습니다.

> "Early in the history of the web, users expected static pages. If you wanted to see new content, you reloaded the page. That was normal at the time, and nobody found it strange."

> "Then, as more sites added real-time features — auto-updating feeds, live notifications, single-page apps — suddenly those older 'reload-only' sites began to feel slow or outdated."

> "The same pattern is happening with local-first apps. As local-first experiences become commonplace — removing spinners, letting users keep working when offline, and syncing in the background — everything else will start to feel frustratingly behind."

옮기자면, Figma, Linear, Notion이 보편화되면서 "즉시 반응"에 대한 사용자 기대치가 그만큼 높아졌다는 뜻입니다.

## Slax Reader가 한 것

Slax Reader는 완전한 Local-first를 구현하지는 않았습니다. 몇 가지 핵심 읽기 동작을 처리했습니다.

**한 것:**

- **하이라이트** — 누르는 순간 색이 나타나고, 로컬에 기록됩니다.
- **댓글** — 입력이 끝나는 순간 로컬 DB에 저장되며, 오프라인에서도 작동합니다.
- **좋아요 / 즐겨찾기** — 토글이 즉시 반영됩니다.
- **보관** — 즉시 완료되며, 목록이 바로 업데이트됩니다.

이 동작들은 온라인, 오프라인, 불안정한 네트워크, 비행기 모드에서 모두 동일하게 동작합니다. 네트워크가 돌아오면 동기화 큐가 모든 것을 서버로 푸시합니다.

**하지 않은 것:**

- **데이터 완전 로컬화** — 본문 데이터는 여전히 주로 클라우드에 있습니다. Slax를 삭제하면 전체 콘텐츠에 로컬로 계속 접근할 수는 없습니다.
- **오프라인 AI Chat** — AI 대화는 클라우드 모델에 의존하므로 오프라인에서는 작동하지 않습니다.

## Slax Reader가 구체적으로 어떻게 만들었나

코드는 모두 [github.com/slax-lab/slax-reader-client](https://github.com/slax-lab/slax-reader-client)에 오픈 소스로 공개되어 있습니다. 아래 스니펫은 이 저장소에서 발췌한 것입니다. Kotlin Multiplatform으로 작성되어 iOS, Android, Web이 같은 local-first 구현을 공유합니다.

### 1. 로컬 저장소: PowerSync SQLite

자체 구현도 아니고, IndexedDB도, Dexie나 RxDB도 아닙니다. Slax Reader는 [PowerSync](https://www.powersync.com/)를 사용합니다. SQLite를 그대로 local-first 동기화 계층으로 만들어 주는 프레임워크입니다.

스키마 일부:

```kotlin
val srUserBookmarkTable = Table(
    name = "sr_user_bookmark",
    trackPreviousValues = TrackPreviousValuesOptions(onlyWhenChanged = true, columnFilter = null),
    columns = listOf(
        Column.text("user_id"),
        Column.integer("is_read"),
        Column.integer("archive_status"),
        Column.integer("is_starred"),
        Column.text("created_at"),
        Column.text("updated_at"),
        // ... plus alias_title, type, deleted_at, starred_at, archived_at, metadata
    )
)

val srBookmarkComment = Table(
    name = "sr_bookmark_comment",
    columns = listOf(
        Column.integer("type"),          // highlights, comments, replies — type tells them apart
        Column.text("user_bookmark_uuid"),
        Column.text("comment"),
        Column.text("approx_source"),
        Column.text("content"),
        Column.text("created_at"),
        Column.text("metadata"),
    )
)

val localBookmarkInfo = Table(
    name = "local_bookmark_info",
    localOnly = true,                    // local-only, never synced
    columns = listOf(
        Column.text("overview"),
        Column.text("key_takeaways"),
        Column.integer("is_downloaded"),
    )
)
```

몇 가지 세부사항:

- 하이라이트, 댓글, 답글이 모두 같은 `sr_bookmark_comment` 테이블을 공유하고, `type` 컬럼으로 구분합니다.
- `trackPreviousValues`는 PowerSync가 각 행의 이전 값을 기록하도록 해, 이후 필드 단위의 충돌 감지가 자연스럽게 가능합니다.
- `localOnly = true`로 표시된 테이블(`local_bookmark_info` 같은)은 기기 로컬에만 존재하고 동기화되지 않습니다. 문서 다운로드 상태, 로컬 요약처럼 기기 간 동기화가 필요 없는 데이터는 여기에 둡니다.

### 2. 먼저 로컬에 쓰고, 나중에 동기화

사용자가 텍스트를 드래그해서 하이라이트가 저장되기까지의 코드 경로 어디에도 "서버 응답을 기다리는" 단계가 없습니다.

```kotlin
// ViewModel: triggered when the user highlights
fun strokeHighlight(webViewState: AppWebViewState, onComplete: (() -> Unit)? = null) {
    webViewState.evaluateJsWithCallback(
        "window.SlaxWebViewBridge.captureCurrentSelection()"
    ) { resultJson ->
        if (resultJson.isNullOrBlank() || resultJson == "null") return@evaluateJsWithCallback
        viewModelScope.launch(Dispatchers.IO) {
            runCatching {
                val data = StrokeCreateData.fromJsResult(resultJson)
                val localId = commentDelegate.addMark(
                    type = MarkType.LINE,
                    source = data.toMarkPath(),
                    approxSource = data.approx_source,
                    selectContent = data.select_content,
                )
                // Returns the moment local write is done; UI refreshes via Flow
                withContext(Dispatchers.Main) { onComplete?.invoke() }
            }.onFailure { /* failure stays local; UI keeps moving */ }
        }
    }
}

// DAO: local SQLite write
suspend fun addMark(
    bookmarkId: String, userId: String, type: MarkType,
    source: List<MarkPathItem> = emptyList(),
    // ... other optional parameters
): String {
    val id = Uuid.random().toString()
    val now = Clock.System.now().toString()
    val metadata = BookmarkCommentMetadata(/* root_id, user_id, parent_id ... */)

    database.writeTransaction { tx ->
        tx.execute(
            """INSERT INTO sr_bookmark_comment
                (id, type, source, user_bookmark_uuid, comment,
                 approx_source, content, is_deleted, created_at, metadata)
                VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)""",
            listOf(
                id, type.value, json.encodeToString(source),
                bookmarkId, comment, json.encodeToString(approxSource),
                json.encodeToString(selectContent), 0, now,
                json.encodeToString(metadata)
            )
        )
    }
    return id   // Returns immediately; caller doesn't wait for network
}
```

UI는 SQLite 테이블 변경을 구독해서 스스로 다시 그립니다.

```kotlin
private val comments: StateFlow<List<BookmarkCommentPO>> = _bookmarkId
    .flatMapLatest { id ->
        id?.let { commentDao.watchComments(it) } ?: flowOf(emptyList())
    }
    .stateIn(scope, SharingStarted.Eagerly, emptyList())
```

`watchComments()`는 PowerSync의 `database.watch()` 위에 작성한 반응형 쿼리입니다. 테이블에 변화가 생기면 Flow가 새 데이터를 푸시하고, UI가 그에 따라 갱신됩니다. 수동 `setState`도 없고, "서버 응답을 기다린 다음 새로 고침" 분기도 없습니다.

### 3. 오프라인 큐: PowerSync 내장 CRUD log

Slax Reader에는 명시적인 `syncQueue` 테이블이 없습니다. PowerSync 자체가 SQLite 내부에 CRUD 작업 로그를 유지하고, 모든 `database.writeTransaction {}`이 자동으로 `CrudEntry`를 기록합니다.

업로드는 `Connector.uploadData()`를 통해 배치로 진행됩니다.

```kotlin
override suspend fun uploadData(database: PowerSyncDatabase) {
    val transactions = mutableListOf<CrudTransaction>()
    val batch = mutableListOf<CrudEntry>()

    database.getCrudTransactions()
        .take(100)          // up to 100 per batch
        .collect { tx ->
            batch.addAll(tx.crud)
            transactions.add(tx)
        }

    if (batch.isEmpty()) return

    try {
        val postData = batch.map { entry ->
            val (changes, preChanges) = diffChanges(entry.opData, entry.previousValues)
            ChangesItem(
                table = entry.table,
                id = entry.id,
                op = entry.op.toString(),    // "INSERT" / "UPDATE" / "DELETE"
                data = changes,
                preData = preChanges
            )
        }
        apiService.uploadChanges(changes = postData)
        transactions.forEach { it.complete(null) }   // success — clear from queue
    } catch (e: Exception) {
        throw e                              // failure — PowerSync auto-retries
    }
}
```

오프라인 상태에서는 이 코드가 실행되지 않습니다. `CrudEntry` 레코드는 SQLite의 CRUD log에 쌓입니다. 네트워크가 돌아오면 PowerSync가 다시 `uploadData()`를 호출하고, 트랜잭션 순서대로 재생합니다.

### 4. 충돌 해결: 타임스탬프 + 필드 단위 diff

전략은 Last-Write-Wins지만, **단순한 전체 행 덮어쓰기는 아닙니다**. 클라이언트가 먼저 필드 단위로 diff를 수행하고, 변경된 필드만 전송합니다.

```kotlin
fun diffChanges(data: SqliteRow?, preData: SqliteRow?)
        : Pair<Map<String, String?>?, Map<String, String?>?> {
    // ...
    for ((key, value) in dataMap) {
        val oldValue = preDataMap[key]
        if (value != oldValue) {
            if (key == "metadata") {
                // JSON field — diff down to sub-fields
                val newMetadataObj = Json.decodeFromString<BookmarkMetadata>(value ?: "")
                val oldMetadataObj = Json.decodeFromString<BookmarkMetadata>(oldValue ?: "")

                if (newMetadataObj.tags != oldMetadataObj.tags) {
                    changes["metadata.tags"] = Json.encodeToString(newMetadataObj.tags)
                    preChanges["metadata.tags"] = Json.encodeToString(oldMetadataObj.tags)
                }
                // share settings get the same sub-field treatment
                // ...
            } else {
                changes[key] = value
                preChanges[key] = oldValue
            }
        }
    }
    return Pair(changes, preChanges)
}
```

실제로는 이렇게 됩니다. 휴대폰에서 tags를 바꾸고, 동시에 다른 기기에서 공유 설정을 바꿔도, **양쪽 변경이 모두 살아남습니다**. 서로 덮어쓰지 않습니다.

전체 구성:

| 데이터 유형 | 전략 |
|---|---|
| 단순 필드(is_read / archive_status / is_starred) | 필드 단위 LWW |
| JSON metadata 하위 필드(tags / share) | 하위 필드 단위 LWW |
| 삭제 | soft delete(`deleted_at`), 충돌은 타임스탬프로 해결 |
| 하이라이트 / 댓글 트리(root_id / parent_id) | 비즈니스 규칙 — 같은 하이라이트에 대한 다른 답글은 공존하며 충돌로 처리하지 않음 |

엔지니어링상의 정직한 현실 하나. `metadata` JSON 파싱이 실패하면(스키마 마이그레이션, 옛 포맷) 코드는 전체 필드 덮어쓰기로 폴백합니다. 이 경우 필드가 손실될 수 있습니다.

### 5. 동기화 트리거: 네트워크 상태 기반

폴링 타이머도 아니고, 실시간 WebSocket도 아니며, **네트워크 상태 변화로 구동됩니다**.

```kotlin
fun startup() {
    workerScope = CoroutineScope(SupervisorJob() + Dispatchers.Default)

    workerScope!!.launch {
        connectivity.statusUpdates.collect { status ->
            if (status is Connectivity.Status.Connected) {
                reconnect()   // network back → reconnect PowerSync → pull + push fire
            } else {
                disconnect()  // network gone → close the sync channel
            }
        }
    }
    // ... two more launch blocks for connect() and syncState
}
```

`Connectivity.statusUpdates`는 웹에서는 `navigator.onLine`을, iOS / Android에서는 대응하는 네이티브 API를 감쌉니다. "Connected"로 전환되는 순간, PowerSync가 새 데이터를 내려받고 로컬 CRUD 큐를 위로 푸시합니다.

앱은 자체 재시도 로직을 쓰지 않습니다. 동기화 책임은 PowerSync로 수렴됩니다. 그 대가는, 특정 동기화 프레임워크에 묶인다는 점이며 경로 의존성이 깊다는 것입니다.

### 엔지니어링상의 타협

RxDB 글은 local-first의 정직한 한계 열 가지를 나열합니다. 그중 두 가지는 브라우저 측 저장소를 사용하는 모두에게 해당됩니다.

> "**Safari notoriously wipes out IndexedDB data if the user hasn't used the site in ~7 days.**"

> "Local-first works best when the data set per user is reasonably sized (**up to 2 Gigabytes**)."

브라우저 로컬 저장소는 장기 저장소로 신뢰할 수 없습니다. Slax가 본문 전체를 로컬에 두지 않는 이유 중 하나이기도 합니다. 본문은 클라우드에 시스템 오브 레코드로 두고, 로컬에는 **사용자가 생성한 상태**(하이라이트, 댓글, 보관 플래그)만 둡니다.
