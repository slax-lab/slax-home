私は UI デザイナーで、コードはずっと別の星の話だと思ってきました。ある日、上司から PM と一緒に AI で [Slax Note](https://slax.com/note/) を書き直してほしいと言われたとき、正直少し怖かったです——*本当に私にできるんだろうか？*

## 1. 最初は、神様だと思っていました

最初の体験は本当に衝撃的でした。エンジニアが Claude Code（以下「彼」と呼びます）の設定を教えてくれて、簡単なトレーニングを受けたあと、私たちは自分で触り始めました。

最初に彼にやってもらったのは、Slax Note の Web 版の有料会員ポップアップの静的ページです。デザイン画を渡したら、彼はそこそこの形に仕上げてくれました。少し調整するように指示すると、ほぼデザイン通りの見た目になりました。あの達成感、「コードってそんなに難しくないのかも！」と思いました。

そのまま本格的にプロジェクトに着手しました。エンジニアのドキュメントを渡すと、フレームワークを組み立て、動かしてみても感触は悪くなく、主要な機能もそろっていました。これで自信がさらにつきました。

## 2. すぐに、彼に「キレ」させられました

### 2.1 とにかく推測したがる

序盤、私は主に UI の再現に集中していました。デザインガイドラインをまとめ、Lanhu MCP（中国のデザイン共有ツール向け MCP）を設定して、Lanhu 上の数値に従って UI を作るように指示しました。それでも彼はデザイン稿に従わず、ひたすら勘で書きました。

あるリンクの色は、「感覚で」グレーにされていました。でもデザイン稿では明らかにオレンジブラウンでした。

余白もそうです。明確な規定がないと、彼は数字を勝手に決めます。私が見てもおかしいので直すよう言う、直してもまだおかしい、何度か往復してからデザイン稿を見直すと、最初からきちんと数値が書かれていたのです。彼が見ていなかっただけでした。

教訓：コードを書かせる前にデザインガイドラインを参照させる、推測させない。推測したら数ラウンドの手戻りが代償です。

もし彼にデザインを細かく説明してあげれば（下のように）、仕様書みたいにガチガチに書けば、ちゃんと作ってくれます。でもこれはデザイナーじゃない人には優しくありません。

> **トップページの録音ボタンの実装説明：** ページ下端から 50pt の位置にカプセル型ボタンが浮かんでいる。ボタンは幅 140pt、高さ 74pt、影あり、深すぎず、見える程度。ボタンの中には左右に丸い小ボタンが 1 つずつ。左の内側 8pt の位置に丸い録音ボタン、サイズ 62pt、背景は上から下に `#04EAC5` から `#00BEC9` のグラデーション、中央に録音アイコン。押下状態ではボタン背景に `#000000` 8% のマスクを重ねる。右の内側 8pt の位置に丸いテキスト入力ボタン、サイズ 54pt、背景 `#FFFFFF`、中央にテキスト入力アイコン。押下状態の背景は `#4D4D4D` 8%。

その後、下の prompt でノートのポスターを生成してみました。位置のセンタリングだけずれましたが、それ以外はデザイン稿どおりに再現されました。これは規定ドキュメントに書き込んでしまえば、毎回書かなくて済みます：

> 設計図に厳密に従って xx ページを再現してください。ピクセル単位で再現し、レイアウト、色、間隔、フォント、アイコン位置を勝手に変更してはいけません。
>
> 以下のルールに従ってください：
>
> 1. 設計図のページ構造、階層関係、レイアウトを完全に複製する
> 2. 色の値、フォントサイズ、間隔、角丸、影は設計図と完全に一致させる
> 3. ボタン、入力欄、カード、ナビゲーションバーなどのコンポーネントの位置とサイズを完全に一致させる
> 4. 設計図にある内容のみを実装し、余計な要素を新規追加しない
> 5. コードは直接動作可能、構造は明瞭、コメントは完全、エラーなし

### 2.2 「ついでに」何かしてくれるのが大好き

小さな Bug を直してと頼むと、彼は周辺のコードもついでに「最適化」してくれて、結果として正常に動いていた機能がいきなり死にます。

それで彼のところに行って直してもらうと、彼はまた直すついでに別のところを触り……。

リフォーム業者に壁の穴を 1 つ塞いでくれと頼んだら、「ついでにこの棚も動かしておきますね、見栄えが良くなるので」と言われ、気がつけば棚の裏のパイプが割れていた、みたいな話です。

その後、彼に 1 つルールを設けました：**コードを触る前に、どこを変更してどこを変更しないつもりかをまず教えてほしい、私が確認してから動いていい**、と。それ以降、だいぶマシになりました。

### 2.3 前回の教訓を覚えていない

これが一番予想外でした。

新しい会話を始めるたびに、彼はまるで記憶喪失です。前回踏んだ落とし穴を、今回も踏みます。

例を挙げます：ポップアップの作り方。

iOS にはポップアップの実装方法が 2 つあります。1 つはシステム標準のもの、もう 1 つは自前で作るフルスクリーンのもの。システム標準のものはポップアップ自体に角丸と余白がついていて、どう調整してもデザイン稿と一致しません。

私は最初に問題に気づいて、直しました。2 回目に新しいページを作ったら、彼はまたシステム標準を使っていました。直す。3 回目もまたシステム標準。また直す。

3 回直したあとに、ようやく気づきました：**これはルールとして書き出して、彼が作業前に毎回見られるようにしないといけない**、と。

この「ルールファイル」が `CLAUDE.md` です。最初は空でしたが、落とし穴を踏むたびに 1 行ずつ追加していき、最後には数百行になりました。

要するに、これは「AI と付き合うためのマニュアル」です。書かなければ、彼は覚えてくれません。

## 3. 本当に心が折れた出来事

### 3.1 左スワイプで削除、10 数ラウンド直した

これは記憶の中で一番修正に時間がかかった機能です。

デザイン稿では、ノートカードを左にスワイプするとアーカイブと削除の 2 つのボタンが出てきます。カスタムの色と角丸付きです。一見普通のインタラクションなので、すぐに作れると思っていました。

ところが、第 1 版が出てきたら、ボタンの色がまったく違いました。直すように言う、直してもまだ違う。あとで分かったのですが、iOS 標準の左スワイプ削除コンポーネントは色がシステム側で「ロック」されていて、何色を指定しても自動的に上書きされてしまうのです——四角い色面と白文字のみ、選択の余地なし。つまり、デザイン稿のあのスタイルは、システムコンポーネントでは作れません。

それなら自前で作るしかない。彼はジェスチャー認識を使って左スワイプのロジックを書き直しました。やっと動くようになった——と思ったら新しい問題：左スワイプのジェスチャーとページの上下スクロールのジェスチャーが競合し、上下にスクロールしたいのに左スワイプと認識されたり、左スワイプしたカードがスワイプアウトしたあと自動的に戻ってきたり。

この「戻ってくる」問題は、前後で 4 ラウンドかかりました。最終的に振り返ると、原因はまったく異なる 4 つの要因が同時に作用していました——ジェスチャー認識のトリガー順序、iOS のシステム更新後の挙動変化、カード間の状態の相互干渉……。一層ずつ修正して、ようやく落ち着きました。

最終的な UI は、実はデザイン稿どおりではなく、彼のステップに合わせて 1 つずつ調整した結果です。動くならスタイルにこだわらない——下手にこだわって新しい問題が出たら……。

教訓：「ただの左スワイプ」に見える機能でも、裏に何層もの問題が潜んでいることがある。1 層直しても、急いで喜ばないこと。

### 3.2 ノートカードをタップすると、アプリがフリーズ

現象：ノートの内容が特に長いとき、そのノートをタップするとアプリが固まって動かなくなる。

Claude は調査を始めて、リストのスクロールジェスチャーやジェスチャーパラメータの問題だと考え、ずっと「ジェスチャー」方向で探っていました。長いこと直し続けても、毎回問題は解決しません。

最終的にどうしようもなくなって、エンジニアの大ベテランを呼びました。彼は別の AI ツール（Codex）を使って独立に調査しました。関連ファイルを 2 分でスキャンして、真の原因を直接突き止めました：

ジェスチャーの問題じゃ、まったくなかった。

実はノート詳細ページの中に「長文の展開／折りたたみ」を担うコンポーネントがあって、レンダリング時に、まず記事全文——5000 字——をバックグラウンドで一度レイアウトして高さを計算してから表示する、という挙動をしていました。この処理がメインスレッドで同期実行されていて、5000 字の計算に数秒かかり、その間アプリ全体が凍りつくのです。

Claude がずっと方向を誤っていたのは、症状が「カードをタップする」という動作で出たため、「タップ」関連のジェスチャーコードの中ばかり探していたからです。でも本当に時間を食っていたのは、まったく別の場所——詳細ページの文字レンダリングコンポーネントで、呼び出しを何階層も挟んだ先にありました。

修正方針：文字コンポーネントに渡す前に切り詰める（カードは先頭 300 字のみ表示、計測は先頭 1200 字までスキャン）。4000 字を超えるノートは自動全文展開をスキップ。詳細ページのデータ読み込みは非同期化して、メインスレッドを止めない。

教訓：AI も間違った方向に「執着」することがある。人（あるいは別のツール）が膠着を打破する必要がある。症状が A で出ても、根因は A→B→C の経路上のどこかに潜んでいる可能性がある。2 〜 3 回直しても効かなかったら、立ち止まって経路全体を最初から追い直すべき。症状の層だけを掘り続けてはいけない。

### 3.3 Apple ログインがエラーに

ある日突然、Apple ID ログイン機能が動かなくなりました。この機能は以前に直したはずなのに、今回も同じエラーが出ました。

彼は調査を始め、行ったり来たり 8 ラウンド直しました。毎回「今度こそ直りました」と自信満々で、毎回起動するとまたエラー。途中でアプリを白画面にしてしまい、白画面の修正にも時間を使いました。最後に過去の修正履歴を彼に渡してようやく、彼は問題の在処を理解しました。

根本原因：コードの問題じゃなかった。Xcode（Apple の開発ツール）の中に「権限スイッチ」があって、それが過去のある自動生成操作でオフになっていたのです。しかもこのスイッチは奥深くに隠れていて、スクリーンショットを見せないと分からないレベルでした。

教訓：問題によってはコードの中にない。先にツールの UI を見に行くべき。

Claude もそのあと賢くなりました——Apple ログイン関連の問題に遭遇したら、まず私にスクリーンショットを送るよう要求するようになり、いきなりコードを直しに行かなくなりました。

### 3.4 Web 版でノートが突然 3 つに

iOS でノートを 1 つ作って、Web 版を見に行ったら、3 つに増えていました。彼は分析を始め、どこかの重複排除ロジックがおかしいと言って数ラウンド直しましたが、解決しませんでした。

最後に彼は直接データベースの生のレコードを見に行って、真相を発見しました：リクエストの送信自体は成功していたのに、応答を受け取るときのパースでエラーが出て、プログラムは失敗したと判断して再送、3 回送って、サーバーは律儀に 3 つノートを作っていたのです。

教訓：すぐにコードを直させるのではなく、まず証拠を探させる——ログを見る、データベースを見る——本当の問題箇所を突き止めてから手を動かす。

### 3.5 アイコンの切り出しがいつも変な見た目に

**アイコンがいつも大きすぎる**：デザイン稿でアイコンは 48px、iOS のコード単位に換算すると 2 で割って 24pt。私は知っているのに、彼は知らないらしい。彼はときどき「画像を私が指定したサイズで表示する」というコードを書き忘れて、アイコンは元のサイズで表示され、通常の倍のサイズで画面の半分を占領します。

**アイコンがいつもぼやける**：原因は切り出しの倍率を間違えていること。@1x / @2x / @3x の 3 つのスロットを一緒に更新していない。間違えて入れる——たとえば @2x の画像を @1x スロットに入れる——と、iPhone はこれを低解像度版と判断し、画面に合わせるために自動で 2 倍に拡大表示するので、アイコンはぼやけます。

**「アイコンを正しいサイズに修正しました」と言うのに、見た目はおかしい**：Lanhu 上では同じアイコンに 2 種類のバージョンがあります。1 つは端ぴったりで切り抜いたもの、もう 1 つは透明の余白を残したもの。私たちが使いたいのは透明余白付きの方ですが、彼はときどき間違えます。間違えると、アイコンが端に貼り付いて、整列がずれて見えます。

そのあとルールに追記しました：アイコンをダウンロードするときは Lanhu の最新版を選ぶ、透明余白付きを優先する、入れる前にピクセルを検証する、@2x スロットに入れる、@1x ではない、切り出しを差し替えるときは 3 つのスロットを必ず確認する、コードでは pt を使う、px は使わない。

## 4. AI は結局、何を省いてくれたのか

正直、「AI を使って、どれくらい速くなった？」と聞かれたら、私の答えはこうです：速くはなった、でも想像していたほどではない。

省けたのは、繰り返しの、機械的なもの：

- 新しいページごとの基本構造、数分で組み立ててくれる
- デザイン稿の色の値、フォントサイズ、間隔の調査と換算
- 中国語・英語・ポルトガル語の 3 言語の翻訳

省けなかったもの：

- 変な bug に遭遇したら、結局人間が 1 つずつ分解する
- 実機テストは AI には頼めない、自分でやるしかない
- 「ここはどの方式を使うべき」みたいな判断は、結局人間が下す

一番正確な比喩はこれだと思います：**AI は実行力が極めて高いけれど、非常に明確な指示を必要とするアシスタント**です。きちんと指示すれば、速く、よくやってくれる。曖昧に指示すれば、彼の解釈で動いてしまい、あとから時間をかけて修正することになる。

## 5. 一番大きな収穫はコードじゃなかった

一ヶ月で、14 ページ完成しました。

正直に言って、AI は本当に記憶力がない。毎日教訓を総括させて記録することを覚えてから、私と彼の「付き合いマニュアル」を数行から数百行に育てて、ようやく彼は「使いやすく」なりました。**私たちが学ぶべきなのはコードの書き方ではなく、AI の指揮の仕方です。**

このどんどん厚くなる「付き合いマニュアル」——踏んだ落とし穴、立てたルール、触ってはいけない場所、すべて書いてあります。

このマニュアルは次のプロジェクトにも持っていけます。AI が私たちの習慣をどんどん「理解」してくれて、無駄な回り道が減ります。
